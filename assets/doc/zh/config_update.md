# 配置文件更新机制

本文从用户角度说明：当你更新 AALC 后，程序文件和配置文件分别会发生什么。

## 简短结论

正常更新不会主动清空你的 `config.yaml`。更新包覆盖的是程序文件、资源文件和默认配置模板；你的实际设置会在下次启动时由程序读取、备份、迁移并重新保存。

如果新版新增了配置项，程序会使用新版默认值补齐。如果旧版配置结构发生变化，程序会按照配置版本号执行迁移逻辑，尽量把旧设置转换为新版可用的格式。

## 更新程序文件时

在程序内检测到新版本后，AALC 会先把更新包下载到 `update_temp` 文件夹。下载完成后，用户确认开始更新，程序会启动 `AALC Updater.exe`。

更新器会执行这些步骤：

1. 解压下载好的更新包
2. 关闭正在运行的 `AALC.exe`
3. 将新版本文件覆盖到当前程序目录
4. 清理下载包和解压目录
5. 重新打开 AALC

如果更新包里没有 `changes.json`，更新器会在覆盖前删除旧的 `assets/images` 文件夹，再复制新版图片资源。这个操作只针对图片资源目录，不是配置文件目录。

## 下次启动时

程序启动后会初始化配置系统。实际使用的配置文件路径是程序目录下的：

```text
config.yaml
```

新版默认配置模板来自：

```text
assets/config/config.example.yaml
```

启动时大致会发生下面这些事：

1. 读取当前 AALC 版本号
2. 读取新版默认配置模板
3. 读取用户已有的 `config.yaml`
4. 先为用户配置创建备份
5. 检查 `config_version`
6. 如果配置版本较旧，执行旧配置迁移
7. 用新版配置模型重新整理配置内容
8. 保存回 `config.yaml`

因此，更新后的第一次启动可能会改写 `config.yaml`。这通常是正常现象，表示程序已经把旧配置转换成新版格式。

## 会生成哪些备份文件

读取 `config.yaml` 时，程序会创建几个备份文件：

| 文件 | 作用 |
| ---- | ---- |
| `config.yaml.bak` | 每次启动读取配置前都会复制一份当前配置 |
| `config.yaml.backup` | 配置成功加载后，再保存一份启动时读取到的配置 |
| `config.yaml.old` | 大约每 6 次成功启动保存一次更早的配置 |

如果更新后发现设置异常，可以先关闭 AALC，再对照这些备份文件恢复。

## 旧配置如何迁移

主配置文件里有一个 `config_version` 字段。新版程序启动时，会比较用户配置中的版本号和当前程序内置的配置版本号。

如果用户配置版本较旧，程序会进入迁移流程。目前代码中包含这些迁移：

| 旧版本条件 | 迁移内容 |
| ---------- | -------- |
| `< 1768403022` | 将旧镜牢历史耗时记录转换为总平均、近 5 次平均、近 10 次平均 |
| `< 1771413380` | 将旧的窗口位置设置转换为新版字符串配置 |
| `< 1771965838` | 将旧的后台点击设置转换为新版输入方式配置 |
| `< 1772205660` | 将旧的“结束后操作”转换为新版组合动作配置 |
| `< 1775826004` | 将旧的 `team1_setting`、`team1_history` 等队伍字段合并到新版 `teams` 结构 |
| `< 1778889600` | 为未启用自选开局加成的旧队伍写入新版默认开局星光 |

这些迁移会尽量保留用户已有设置。对于已经存在的新版字段，程序通常不会用旧字段覆盖它们。

## 新增配置项如何补齐

保存配置时，程序会先读取新版 `config.example.yaml`，再把用户当前配置覆盖进去。

这样做有两个效果：

1. 新版新增的字段会从默认配置模板中出现
2. 用户已经设置过的字段会继续使用用户值

也就是说，更新后你可能会在 `config.yaml` 里看到新的字段、注释或顺序变化，这通常是正常的。

如果旧配置里有已经废弃、且新版配置模型不再认识的字段，它们可能会在重新保存后消失。这类字段不再参与新版程序运行。

## 队伍配置会怎样

新版队伍设置主要保存在 `config.yaml` 的 `teams` 字段中。旧版本中类似 `team1_setting`、`team1_history`、`team1_remark_name` 的字段，会在迁移时合并进 `teams`。

程序还会维护队伍启用顺序：

- 新版字段：`teams_active_queue`
- 兼容旧逻辑的字段：`teams_be_select`、`teams_order`、`teams_be_select_num`

启动时程序会规范化队伍队列，移除无效队伍编号和重复编号，然后同步回这些字段。

## 主题包权重配置会怎样

主题包权重配置和主配置分开维护：

```text
theme_pack_list.yaml
theme_pack_weight/theme_pack_weight_team_*.yaml
```

启动时程序会读取新版默认主题包模板：

```text
assets/config/theme_pack_list.example.yaml
```

如果 `theme_pack_list.yaml` 不存在，程序会自动创建。

如果它已经存在，程序会用新版模板补齐缺失内容，同时保留用户已有的权重设置。已有的队伍自定义权重文件也会按同样方式同步。

## 什么情况需要手动处理

一般不需要手动处理配置文件。下面几种情况才建议查看备份：

1. 更新后程序提示 `config.yaml` 加载错误
2. 更新后某些设置明显丢失或异常
3. 手动编辑过配置文件，并写入了新版无法识别的格式
4. 手动把程序解压到了旧目录，并覆盖了不该覆盖的文件

恢复时建议优先查看 `config.yaml.bak` 或 `config.yaml.backup`。如果只是想回到更早的状态，再查看 `config.yaml.old`。

## 代码位置

相关逻辑主要在这些文件中：

- 程序更新覆盖流程：`updater.py`
- 下载更新包并启动更新器：`module/update/check_update.py`
- 主配置初始化、备份、迁移、保存：`module/config/config.py`
- 主配置字段定义和当前配置版本号：`module/config/config_typing.py`
- 配置文件路径常量：`module/__init__.py`
- 队伍设置导入导出：`module/config/team_import_export.py`
